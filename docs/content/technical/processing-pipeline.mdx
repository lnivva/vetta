# Processing Pipeline

The STT service is a standalone **streaming gRPC server** that owns everything between raw audio and transcribed segments.

The core library sends audio in and receives **streamed, timestamped transcript chunks** back. It has no knowledge of the models, diarization pipeline, or runtimes involved.

![STT Service Architecture](/vetta-processing-pipeline-arch.png)

## Stages

Every request follows the pipeline below. Some stages may run **in parallel** or be **skipped** depending on request and configuration.

1. **Resolve**: Determine the audio source (local path, inline bytes, or remote URI). Remote URIs are validated against SSRF rules, fetched with size limits, and validated before decoding.
2. **Preprocess**: Decode audio and prepare model-specific inputs. Audio is normalized for Whisper inference, and if diarization is requested, a separate seekable input is prepared for the diarization pipeline.
3. **Diarize (optional)**: Run speaker diarization to produce timestamped speaker labels. This stage is optional, configuration-driven, and lazily initialized. When enabled, diarization may execute in parallel with transcription.
4. **Transcribe**: Run Whisper inference to produce timestamped segments (and optionally word-level timestamps). Segments are collected before downstream processing.
5. **Post-process & Stream**: Apply speaker attribution, optional segment stitching, entity correction, punctuation, and truecasing, then stream `TranscriptChunk` messages back to the caller over gRPC.

## Transport

The server supports multiple gRPC transport options depending on deployment:

- **Unix Domain Socket**: default for same-host deployments to avoid TCP overhead
- **TCP**: for cross-machine or containerized setups

Transport is a configuration choice; the processing pipeline is identical regardless of transport.

## Speaker Attribution

Diarization and transcription run as **independent inference passes**. Whisper produces timestamped text segments (and words when enabled); the diarization pipeline produces timestamped speaker turns.

Speaker labels are assigned by **maximum temporal overlap**:
- Each segment is assigned the speaker with the greatest overlap
- Word-level speaker labels are assigned using the same overlap logic

If diarization is unavailable or disabled, segments are emitted without speaker labels.

This design allows the transcription model, diarization model, and post-processing logic to be upgraded or replaced independently.

## Configuration

All runtime behavior (model selection, device placement, compute type, concurrency, diarization enablement, and post-processing features) is driven by configuration with environment variable overrides.

Diarization dependencies are loaded lazily, allowing the service to start even when diarization support is not installed. Optional components fail gracefully unless explicitly marked as required.
